技術細節說明

高併發防超賣

後端採用 JPA 樂觀鎖 @Version 驗證。在高頻寫入時，若偵測到版本衝突，系統會拋出並處理 OptimisticLockingFailureException。

Pinia 15分鐘倒數

前端點擊鎖定座位後，Pinia 狀態機開啟倒數計時，同步將座位臨時保留。若超時未發起付款，系統將觸發排程回收機制。

介面冪等性防護

下單時前端生成唯一的 UUID submitToken。後端調用 Redis 設置具有過期時間的 Key，作為分散式鎖，攔截 3 秒內的重複點擊請求。

JPA Proxy 優化

執行 orderTicket 時使用 entityManager.getReference() 載入關聯實體，避免無謂的 JOIN 查詢，顯著降低高頻寫入下的資料庫負荷。

SHA-256 簽章加密

基於綠界 API 規範，將所有交易請求參數按字母排序，串接 HashKey 與 HashIV 後進行 URL 編碼，最終以 SHA-256 加密生成 CheckMacValue。

動態跳轉 Form 表單

後端處理完金流參數後，動態產生內嵌 Auto-Submit JS 的 HTML Form 回傳給前端，達到無感重新導向至金流付款頁面。

可靠 Webhook & 冪等性

提供 S2S Webhook 回呼。後端嚴格驗證綠界回傳的簽章，解析交易單號前綴（如 Ticket_ 或 Goods_）路由對應業務。修改資料庫狀態後，回傳 1|OK 回應以阻斷後續的重發機制。

前端 QR Code 掃碼

調用 html5-qrcode 套件取得本機相機存取權，偵測 QR Code 條碼並即時解密提取雜湊簽章，免除原生 APP 的依賴。

三防核銷機制 (3-Defense)

1. 防無效：驗證資料庫中雜湊值是否存在與合法。
2. 防退票：狀態必須為 PAID，且未申請退款。
3. 防多用：校驗 is_used 狀態，已核銷則直接拒絕。

即時原子操作

核銷通過時，於資料庫事務中原子更新 is_used = true 且記錄 checked_in_at，有效杜絕高併發下的重複核銷漏洞。

SaaS 自動分帳

主辦方設定抽成比例（例：5%）。SettlementService 會依結算週期彙整已核銷訂單，自動分流平台抽成和主辦方應得之淨利潤。

營收看板與趨勢

前端串接 Chart.js 與後端營收統計接口，動態渲染日營業額、月趨勢、退票率等商業指標，提供主辦方一目了然的儀表板。

提款審核閉環

主辦方可於後台申請提款。系統產生付款單後，需由平台管理員手動核對帳務資訊並點擊審核通過，串接撥款以確保資金安全性。
